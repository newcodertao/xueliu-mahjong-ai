# 血流麻将策略与 UI 优化开发计划

日期：2026-07-17  
项目：`xueliu-mahjong-ai`  
阶段：区域识别基本稳定后的策略与交互优化

## 实施状态（2026-07-17）

首轮工程实现已完成：

- 已建立统一的 `StrategyContext`、`StrategyAction`、`ActionEvaluation` 和评分分解。
- 已扩展血流规则、番型、结算与策略权重配置。
- 已实现确定性出牌 V2、实际剩余进张、听牌质量和两步改良。
- 已升级换三张、定缺、碰、杠、胡决策接口。
- 已实现分玩家基础风险模型和进攻/防守综合评分。
- 已实现固定种子、合法剩余牌池、隐藏手牌采样和时间预算蒙特卡洛基线。
- 已建立带 LRU 缓存的统一策略引擎并接入实时 UI。
- UI 候选区已显示总分、向听、进张、两步改良和风险。
- 实时日志已记录策略版本和计算耗时。
- 已增加策略金标准执行器、日志回放报告和训练数据版本化接口。
- 已增加 `strategy-report` 和 `strategy-gold` CLI。

仍需依靠真实数据持续完成的内容：

- 按实际客户端确认全部番型、封顶、胡后手牌变化和结算细节。
- 人工审核并积累不少于 100 个策略金标准案例。
- 从高质量实战牌谱学习风险权重、动作先验和状态价值。
- 使用完整血流规则模拟器进行大规模自对弈训练。

上述项目不能使用自动生成标签替代真实规则确认或人工策略审核，否则会把当前启发式策略自身固化为“金标准”。

## 1. 目标

在现有“截图 → 识别 → 结构化牌桌 → 推荐门控 → UI”链路上，建立一套可解释、可回放、可测试、可逐步学习的血流麻将策略系统，并让 UI 清楚展示推荐结论、依据、风险和识别可信度。

本阶段不直接训练端到端强化学习模型。优先完成确定性策略、规则配置、动作期望价值、风险模型和离线回测；数据积累充分后再引入蒙特卡洛与学习模型。

## 2. 当前基础

### 2.1 已具备

- 27 种序数牌编码和牌数校验。
- 普通牌型、七对向听数与胡牌判断。
- 有效进张计算。
- 定缺花色强制出牌。
- 基础换三张、定缺、碰、杠、胡建议。
- 牌桌可见牌统计。
- 结构化手牌、副露、弃牌和阶段信息。
- 推荐就绪度和识别质量门控。
- JSONL 对局日志、视频复盘和金标准测试框架。

### 2.2 主要不足

- 出牌评分主要是“向听数 + 一步进张 + 简单牌形权重”。
- 未评价听牌质量、两步改良、预期番数和血流连续胡牌价值。
- 碰牌只比较碰前碰后向听数。
- 杠牌基本只判断是否具备杠牌条件。
- 防守模型只根据已见张数计算简单风险。
- `monte_carlo.py` 尚未实现真实模拟。
- 血流规则、番型和结算细节没有形成完整的可配置规则档案。
- UI 没有统一展示候选动作差距、收益、风险和推荐置信度。

## 3. 总体设计

### 3.1 分层架构

```text
StructuredTableState
        ↓
StrategyContextBuilder
        ↓
LegalActionGenerator
        ↓
DeterministicEvaluator
        ↓
RiskEvaluator / ScoreEvaluator
        ↓
MonteCarloEvaluator（后续阶段）
        ↓
ActionRanking + Explanation
        ↓
Realtime UI / Replay / JSONL
```

视觉层只负责提供事实和不确定性，不直接决定策略。策略层只能消费结构化状态，不能读取截图或 YOLO 检测对象。

### 3.2 统一策略上下文

新增统一数据结构 `StrategyContext`，至少包含：

- 当前暗手牌。
- 自己及三名对手的副露。
- 四名玩家的弃牌。
- 已见牌计数和剩余牌估计。
- 当前玩家定缺花色。
- 能够推断出的对手定缺花色。
- 当前牌局阶段和轮次。
- 牌墙剩余量或估计值。
- 当前番型、向听数和已开门数量。
- 识别质量、推断牌数量和未知信息。
- 血流模式下已胡次数及结算上下文。

策略函数不得继续传递大量零散参数。旧接口可以保留为兼容包装层。

### 3.3 统一动作模型

新增 `StrategyAction`：

- `discard`
- `exchange_three`
- `choose_missing_suit`
- `peng`
- `ming_kong`
- `an_kong`
- `bu_kong`
- `hu`
- `pass`

新增 `ActionEvaluation`：

- 动作和目标牌。
- 总分和排序。
- 动作前后向听数。
- 一步有效进张。
- 两步改良率。
- 胡牌概率估计。
- 预期得分和净收益。
- 分玩家点炮风险。
- 牌型破坏成本。
- 不确定性惩罚。
- 推荐理由、警告和关键证据。
- 与最佳动作的分差。

## 4. 动作价值模型

第一版统一采用可解释的加权模型：

```text
动作价值 =
    即时确定收益
  + 胡牌概率 × 预计胡牌收益
  + 有效进张价值
  + 两步改良价值
  + 血流连续胡牌价值
  + 杠牌收益
  - 点炮概率 × 预计损失
  - 查花猪风险
  - 查大叫风险
  - 牌型破坏成本
  - 信息不确定性惩罚
```

所有权重进入配置文件，不在算法中散落硬编码。每次策略日志记录实际使用的权重版本。

## 5. 分阶段开发

## 阶段 A：规则和基础模型重构

### A1. 血流规则档案

扩展 `configs/rule_xueliu.yaml`：

- 换三张方向和限制。
- 定缺约束。
- 是否允许吃牌。
- 胡牌后是否继续。
- 胡后手牌变化规则。
- 自摸、点炮、杠上花、抢杠胡等结算。
- 刮风下雨规则。
- 查花猪、查大叫规则。
- 支持的番型、番数、封顶方式。
- 客户端或房间规则变体。

### A2. 计分引擎

- 完善番型识别。
- 支持开放副露下的番型判断。
- 输出番型明细，不只输出总番数。
- 对规则变体增加参数化测试。
- 对未知或暂不支持的番型明确降级，不能静默按零分处理。

### A3. 策略接口

- 新增 `strategy/context.py`。
- 新增 `strategy/actions.py`。
- 新增 `strategy/evaluation.py`。
- 让现有 advisor 逐步迁移到统一接口。

### 验收标准

- 同一结构化状态可被实时 UI、视频回放和单元测试重复调用。
- 策略层不依赖 OpenCV、YOLO、Tkinter。
- 所有规则差异均来自配置或明确的规则对象。

## 阶段 B：精确牌效和出牌策略

### B1. 一步牌效修正

- 对每个合法弃牌计算准确向听数。
- 计算有效牌种类和实际剩余张数。
- 扣除手牌、副露和弃牌中的已见牌。
- 避免重复牌种重复累计。
- 输出每种有效牌及剩余张数。

### B2. 听牌质量

- 区分单骑、边张、嵌张、两面和多面听。
- 统计有效听口数量和剩余总张数。
- 结合对手副露与弃牌修正实际可获得概率。
- 识别“形式听牌但有效牌已接近耗尽”的情况。

### B3. 两步改良

- 对非听牌状态枚举下一张可能摸牌。
- 计算进入更低向听或更高有效张数状态的概率。
- 设置计算预算和缓存，保证实时 UI 可用。
- 相同手牌状态使用规范化键缓存结果。

### B4. 牌型与血流价值

- 比较快速小胡与高番慢胡。
- 评价对子、刻子对连续胡牌的保留价值。
- 引入清一色、七对、对对胡等发展潜力。
- 牌墙末段提高听牌和即时胡牌权重。

### 验收标准

- 相同向听下能正确区分实际进张多寡。
- 推荐结果能说明“为什么第一选择优于第二选择”。
- 建立不少于 100 个出牌金标准案例。
- 所有案例可批量回测并输出差异报告。

## 阶段 C：换三张和定缺联合优化

换三张和定缺不能完全独立。换牌时应同时估计换牌后最可能选择的定缺花色。

### 开发内容

- 枚举所有合法的同花色三张组合。
- 对每个组合计算换出前手牌结构损失。
- 对收到未知三张牌建立概率分布。
- 评估换牌后各定缺选择的期望价值。
- 避免换出对子、完整顺子和高价值搭子，除非花色必须舍弃。
- UI 展示推荐三张、备选组合和建议定缺。

### 验收标准

- 不再仅按边张优先级选择换牌。
- 对同一手牌可以解释换牌和定缺之间的联动。
- 使用规则生成和人工整理案例覆盖极端花色分布。

## 阶段 D：碰、杠、胡和过牌决策

### D1. 碰牌

比较：

```text
不碰后的期望价值
vs
碰牌并执行最佳弃牌后的期望价值
```

评价因素：

- 向听数和实际有效张数变化。
- 是否破坏七对或高价值搭子。
- 是否增加对对胡、清一色潜力。
- 碰后被迫打出的牌及其危险度。
- 手牌灵活性损失。
- 当前巡目和牌墙剩余量。

### D2. 杠牌

分别处理暗杠、直杠和补杠：

- 杠前杠后向听和听口变化。
- 抢杠风险。
- 杠后摸牌收益。
- 杠上花收益。
- 刮风下雨即时收益。
- 牌墙末段价值。
- 血流继续胡牌价值。

### D3. 胡牌

- 校验定缺和牌型合法性。
- 计算当前胡牌收益。
- 若具体血流规则允许选择继续策略，评估立即胡与后续收益。
- 不允许仅根据牌型完成就无条件输出建议。

### 验收标准

- 每个碰杠决策都存在明确的“不操作”对照项。
- 杠牌不会破坏合法听牌或产生非法手牌状态。
- 所有动作都能输出前后状态对比。

## 阶段 E：防守和对手建模

### E1. 分玩家风险

为每张候选弃牌输出：

```text
left_risk
top_risk
right_risk
combined_risk
```

### E2. 第一版风险特征

- 已见张数和剩余张数。
- 对手定缺花色。
- 对手副露牌及花色集中度。
- 对手弃牌时间顺序。
- 早巡、中巡、末巡。
- 对手疑似清一色或对对胡。
- 对手是否可能已经听牌。
- 某牌是否为明确安全牌、相对安全牌或高危牌。

### E3. 进攻与防守切换

- 自己向听数。
- 自己预计番数。
- 对手威胁等级。
- 当前积分或结算目标。
- 牌墙剩余量。
- 最安全牌与最高牌效牌之间的收益差。

### 验收标准

- UI 能显示风险来自哪个玩家以及主要原因。
- 风险模型不能使用视觉层未确认的信息作为确定事实。
- 缺少对手信息时返回低置信度，而不是虚构精确风险。

## 阶段 F：蒙特卡洛期望值

### F1. 剩余牌池

- 从完整 108 张序数牌中扣除所有可靠可见牌。
- 推断牌只按置信度参与，不直接当作绝对事实。
- 检查任何牌不超过 4 张。

### F2. 隐藏信息采样

- 为对手采样满足牌数约束的暗手牌。
- 根据定缺、副露和弃牌历史调整条件概率。
- 保证所有采样状态合法。

### F3. 前向模拟

- 为每个候选动作执行固定次数模拟。
- 后续动作使用确定性基础策略。
- 支持固定随机种子。
- 支持计算预算、超时和提前停止。
- 输出均值、方差和置信区间。

### F4. 模拟指标

- 胡牌率。
- 自摸率。
- 点炮率。
- 平均净收益。
- 听牌率和平均听牌巡数。
- 查花猪、查大叫风险。
- 多次胡牌期望。

### 验收标准

- 固定状态和随机种子得到可重复结果。
- 模拟不会生成超过 4 张同牌的非法状态。
- UI 线程不被长时间计算阻塞。
- 模拟结果与确定性策略差异可被记录和回放。

## 阶段 G：数据学习

在积累足够可靠的牌局状态、动作和最终收益后执行：

1. 用高质量牌谱训练动作模仿模型。
2. 训练状态价值或动作价值预测模型。
3. 用规则模拟器进行自对弈强化。
4. 使用模型生成动作先验，再用蒙特卡洛验证。

训练数据必须记录：

- 规则版本。
- 状态特征版本。
- 动作集合。
- 实际动作。
- 每次胡牌、杠牌和最终净收益。
- 识别质量。
- 是否包含推断牌。
- 数据来源是实战、视频、人工标注还是模拟。

低质量视觉状态不能直接进入策略训练集。

## 6. UI 优化计划

## 6.1 信息层级

实时界面分为四层：

1. 核心动作：当前建议做什么。
2. 核心原因：向听、进张、收益和风险。
3. 候选对比：前 3～5 个动作的差异。
4. 调试详情：识别状态、结构化牌桌和全部特征。

默认界面不应把调试信息与操作建议混在一起。

## 6.2 推荐主卡片

主卡片显示：

- 当前阶段。
- 推荐动作。
- 推荐牌图形或大号文字。
- 推荐强度：明确、轻微优势、接近、暂缓。
- 与第二选择的差值。
- 一句话原因。
- 数据可信度和是否使用推断信息。

示例：

```text
推荐：打 9万
优势：明显（比 1条高 1.8 分）
原因：保持一向听，有效牌 22 张，三家综合风险低
模式：牌桌增强 / 核心状态可靠
```

## 6.3 候选动作表

建议列：

- 动作。
- 总分。
- 向听数。
- 有效张数。
- 预计番数。
- 胡牌率。
- 点炮风险。
- 主要理由。

支持在“简单”和“专家”视图间切换。

## 6.4 风险可视化

- 在候选弃牌旁显示绿、黄、红风险等级。
- 分别显示上家、对家、下家风险。
- 高风险必须给出文本原因。
- 不确定风险使用灰色或虚线，不能伪装成精确结论。

## 6.5 牌桌与手牌可视化

- 在预览图上高亮推荐弃牌。
- 以不同样式标记观测牌、历史恢复牌和低置信度牌。
- 点击候选动作时预览动作后的手牌结构。
- 支持显示有效牌及剩余张数。

## 6.6 状态和降级提示

UI 明确区分：

- 暂停推荐。
- 仅手牌策略。
- 牌桌增强策略。
- 蒙特卡洛增强策略。
- 使用推断牌的谨慎推荐。

识别不完整时继续提供安全的基础信息，但不能显示虚假精确的收益率。

## 6.7 性能

- UI 刷新和策略计算分离。
- 确定性策略目标耗时小于 50 ms。
- 两步牌效目标耗时小于 150 ms。
- 蒙特卡洛异步执行，先显示快速建议，再更新增强建议。
- 状态没有变化时复用策略结果。

## 7. 日志和回测

每次策略决策记录：

- 状态哈希。
- 规则和策略版本。
- 完整合法动作集合。
- 每个动作的评分分解。
- 最终推荐。
- 推荐模式和可信度。
- 计算耗时。
- 蒙特卡洛随机种子和模拟次数。
- 用户实际动作（能够可靠识别时）。
- 后续胡牌、点炮和净收益。

新增离线报告：

- 推荐覆盖率。
- 暂停推荐比例及原因。
- 第一、第二候选平均分差。
- 推荐与实际动作一致率。
- 不同策略版本的收益对比。
- 识别错误与策略错误分离统计。

## 8. 测试体系

### 8.1 单元测试

- 牌数、规则、向听、进张、番型和计分。
- 所有动作合法性。
- 动作前后状态守恒。
- 剩余牌不超过 4 张。
- 固定随机种子的可重复性。

### 8.2 金标准策略案例

案例分类：

- 强制定缺。
- 相同向听不同进张。
- 好型与差型听牌。
- 碰与不碰。
- 杠与不杠。
- 七对和对对胡冲突。
- 高番进攻与安全防守。
- 牌墙末段。
- 查花猪和查大叫。
- 血流连续胡牌。

每个案例记录允许动作集合、推荐动作、可接受备选以及人工解释。

### 8.3 回归门槛

- 所有既有测试必须通过。
- 策略金标准不能出现未审核退化。
- 关键动作合法性必须达到 100%。
- 性能超过预算时测试失败。
- 不确定状态下不能绕过推荐就绪度门控。

## 11. 首轮开发任务

建议立即执行以下任务：

1. 明确实际客户端的血流计分和胡后规则，补齐规则配置。
2. 创建 `StrategyContext`、`StrategyAction` 和 `ActionEvaluation`。

4. 实现精确有效牌明细和剩余张数。
5. 实现候选动作评分分解。
6. 建立首批 30～50 个出牌金标准案例。
7. UI 先接入新评价结果，但暂不大改整体布局。

完成上述任务后，再进入两步牌效、碰杠和风险模型开发。

## 12. 完成定义

本阶段最终完成需同时满足：

- 推荐基于统一策略上下文，不再由 UI 拼装零散参数。
- 所有合法动作可以统一评价和排序。
- 推荐原因可以追溯到具体评分项。
- 策略支持实时运行、离线回放和批量回测。
- UI 能清楚区分识别质量、策略模式、动作收益和风险。
- 策略错误与视觉错误可以分别定位。
- 蒙特卡洛结果可重复且不阻塞 UI。
- 规则、权重和策略版本完整记录。
- 为后续监督学习和强化学习留下稳定的数据接口。
